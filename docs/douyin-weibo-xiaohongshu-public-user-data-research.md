# 抖音、微博、小红书公开用户数据调研

最后更新：2026-06-12  
调研口径：不以开放平台为判断依据，只看公开网页、移动网页、社区采集实践和当前项目可落地性。  
目标字段：用户名称、头像、粉丝数、关注数、作品/内容数、获赞/互动等公开资料。

## 1. 结论摘要

| 平台 | 名称/头像 | 粉丝数 | 关注数 | 作品/内容数 | 任意指定用户采集可行性 | 建议 |
| --- | --- | --- | --- | --- | --- | --- |
| 微博 | 可获取概率较高 | 可获取概率较高 | 可获取概率较高 | 可获取概率较高 | 中 | 优先做低频 PoC |
| 抖音 | 可获取 | 可获取 | 可获取 | 可获取 | 低到中 | 暂不进生产采集链路 |
| 小红书 | 可获取 | 可获取 | 可获取 | 可获取 | 低 | 暂缓接入 |

一句话判断：

- **微博** 最接近当前 B站公开接口模式，但现在也常受访客系统、Cookie、入口页会话影响。
- **抖音** 字段本身能看到，但 Web 采集高度依赖动态参数、Cookie、设备环境和签名维护。
- **小红书** 字段本身能看到，但稳定采集通常依赖浏览器上下文、登录态和签名参数，不适合作为首期后端定时采集。

## 2. 本地轻量验证记录

测试日期：2026-06-12。  
测试方式：普通 `curl` / PowerShell 请求，不携带账号 Cookie，不使用浏览器自动化，不做签名生成，不尝试绕过风控。

| 平台 | 测试入口 | 观察结果 |
| --- | --- | --- |
| 微博 | `weibo.com/ajax/profile/info?uid=...` | 返回 `Sina Visitor System`，说明匿名脚本请求可能被访客系统拦截。 |
| 微博 | `m.weibo.cn/api/container/getIndex?...` | 出现 `404`、空响应或异常响应，入口参数和访客会话影响较大。 |
| 抖音 | `www.douyin.com` / 用户页形式 | 能返回站点响应和设备 Cookie，但用户详情不是简单裸接口。 |
| 小红书 | `www.xiaohongshu.com/user/profile/...` | 对样例用户页返回 `404`/访问控制相关响应，匿名脚本环境不稳定。 |

这个验证说明：三家都不是“服务端定时器直接拼 URL 就能长期稳定拿数据”的类型。

## 3. 微博

### 3.1 可获取字段

社区爬虫项目长期支持微博用户资料采集，常见字段包括：

- 昵称。
- 头像。
- 粉丝数。
- 关注数。
- 微博数。
- 简介。
- 认证信息。
- 主页地址。

其中 `dataabc/weiboSpider` 明确以用户维度爬取微博数据，输出用户信息和微博信息；其衍生说明中也列出昵称、微博数、关注数、粉丝数等字段。

### 3.2 可能的数据来源

| 来源 | 字段 | 稳定性 | 说明 |
| --- | --- | --- | --- |
| 移动端接口 `m.weibo.cn/api/container/getIndex` | 用户资料、粉丝/关注/微博数、内容列表 | 中 | 需要正确 container、Referer、访客状态；匿名环境不总是稳定。 |
| Web Ajax 接口 `weibo.com/ajax/profile/info` | 用户资料 | 中低 | 容易遇到 `Sina Visitor System`。 |
| 移动网页 HTML | 基础展示字段 | 中低 | 解析成本较高，页面结构变化会影响。 |
| 社区爬虫方案 | 用户信息 + 微博列表 | 中 | 已有成熟实践，但仍要面对反爬、Cookie 和频率问题。 |

### 3.3 风控和限制

- 匿名脚本请求可能进入访客系统。
- 不同用户、不同时间、不同 IP 的返回稳定性不一致。
- 内容列表、评论等比基础资料更容易触发限制。
- 登录 Cookie 可提高稳定性，但当前项目不建议默认引入账号态。

### 3.4 工程判断

微博可以作为第二个平台做 PoC，但要按“不保证每次可用”的模型设计：

- 低频采集，例如 6 小时或 24 小时。
- 只采集公开基础资料和粉丝/关注/微博数。
- 失败后进入退避，不连续重试。
- 不把 Cookie 作为默认必需项。
- 如果后续允许用户自带 Cookie，必须加密存储、脱敏日志、可撤销。

推荐优先级：**P0 PoC**。

## 4. 抖音

### 4.1 可获取字段

公开用户页和社区采集项目通常能拿到：

- 昵称。
- 头像。
- 粉丝数。
- 关注数。
- 获赞数。
- 作品数。
- 用户简介。
- 作品列表和部分作品互动指标。

字段本身不是主要问题，主要问题是请求链路稳定性。

### 4.2 可能的数据来源

| 来源 | 字段 | 稳定性 | 说明 |
| --- | --- | --- | --- |
| Web 用户详情接口 | 用户资料、粉丝/关注/获赞、作品数 | 低到中 | 通常依赖 `sec_user_id`、动态参数、Cookie、设备环境。 |
| 用户主页 HTML / SSR 数据 | 基础资料 | 低到中 | 页面结构和首屏数据策略变化较快。 |
| 社区工具 API | 用户详情、作品、评论、直播等 | 中 | 依赖工具维护签名和运行环境。 |
| 浏览器自动化上下文 | 用户资料和公开内容 | 中 | 成本高，不适合轻量后端定时任务。 |

### 4.3 风控和限制

抖音 Web 采集常见依赖包括：

- `msToken`、`ttwid`、`webid` 等 Cookie 或设备状态。
- 与请求 URL、User-Agent、浏览器环境相关的动态签名参数。
- 真实浏览器上下文或模拟环境。
- 频率、IP、设备指纹一致性。

这些依赖会让 Spring Boot 后端直接定时采集的维护成本很高。社区项目中有生成 Web API 动态参数的实现，也有基于 Playwright 的方案，但这类实现不适合直接作为当前项目的默认生产能力。

### 4.4 工程判断

抖音可以做研究态能力，但不建议第一阶段接进通用采集链路：

- 不默认创建后端定时任务。
- 不实现或内置签名生成逻辑。
- 可以做“手动验证账号是否可读”的实验页面。
- 如果必须做，建议隔离成独立 Worker，开启 feature flag，限制样本数量和频率。
- 数据新鲜度要标注为“不稳定来源”。

推荐优先级：**P1 研究态，不进首期生产**。

## 5. 小红书

### 5.1 可获取字段

小红书用户主页公开展示或社区采集项目通常能拿到：

- 昵称。
- 头像。
- 粉丝数。
- 关注数。
- 获赞与收藏数。
- 笔记数。
- 简介。
- 笔记列表和部分互动指标。

和抖音类似，字段不是核心障碍，核心障碍是稳定采集链路。

### 5.2 可能的数据来源

| 来源 | 字段 | 稳定性 | 说明 |
| --- | --- | --- | --- |
| Web 用户主页 | 基础资料、粉丝/关注、获赞收藏 | 低 | 匿名脚本环境不稳定，容易遇到访问控制。 |
| Web API | 用户资料、笔记列表 | 低到中 | 通常依赖签名参数、Cookie、浏览器上下文。 |
| 浏览器自动化 + 登录态 | 用户资料、笔记、评论 | 中 | 工程成本高，需要维护登录态。 |
| 社区工具 | 多平台公开信息采集 | 中 | 多数明确依赖 Playwright、登录态缓存、代理池等。 |

### 5.3 风控和限制

小红书采集实践里常见依赖：

- Playwright 或类似浏览器自动化。
- 二维码、手机号、Cookie 等登录方式。
- 保存浏览器上下文登录态。
- 从浏览器环境获取签名参数。
- 代理池和低频策略。

这说明小红书不适合作为当前项目的“纯后端 HTTP Adapter”。如果引入，会明显改变系统复杂度和风险模型。

### 5.4 工程判断

小红书首期不建议接入：

- UI 中可以保留“小红书待接入/需浏览器会话”的平台状态。
- 不做默认采集任务。
- 不保存用户级明细，除非产品明确需要并完成合规评估。
- 后续如果要做，只能作为独立 Worker + 浏览器会话方案，不应塞进 Spring Boot 主进程。

推荐优先级：**P2 暂缓**。

## 6. 对当前项目的接入建议

### 6.1 平台优先级

| 优先级 | 平台 | 原因 |
| --- | --- | --- |
| P0 | 微博 | 字段可得性最高，社区实践成熟，适合低频 PoC。 |
| P1 | 抖音 | 字段可得，但动态参数和风控成本高，先做研究态。 |
| P2 | 小红书 | 浏览器会话和登录态依赖重，暂缓。 |

### 6.2 建议新增能力分级

不要把这三家直接标成“已接入”。建议在系统里定义来源等级：

| 等级 | 含义 | UI 展示 |
| --- | --- | --- |
| `PUBLIC_STABLE` | 公开接口较稳定，适合定时采集 | 可正常启用 |
| `PUBLIC_FRAGILE` | 公开可见但接口不稳定 | 显示“实验来源” |
| `BROWSER_ASSISTED` | 需要浏览器上下文或登录态 | 显示“需浏览器会话” |
| `UNAVAILABLE` | 暂不可稳定获取 | 显示“待接入” |

当前建议：

- 微博：`PUBLIC_FRAGILE`
- 抖音：`BROWSER_ASSISTED`
- 小红书：`BROWSER_ASSISTED`

### 6.3 建议采集字段模型

第一阶段只做账号资料快照：

```text
platform_code
external_id
display_name
avatar_url
profile_url
follower_count
following_count
content_count
liked_count
raw_payload_hash
captured_at
source_type
source_status
last_error_type
```

不要第一阶段就做评论、粉丝列表、私域关系链、批量搜索。

### 6.4 建议实现顺序

1. 新建 `external_profile_probe` 或复用 `subject_account_monitor` 增加 `source_type` 字段。
2. 做微博 PoC：
   - 输入 UID 或主页 URL。
   - 低频抓取基础资料。
   - 失败记录 `VISITOR_BLOCKED`、`NOT_FOUND`、`RATE_LIMITED`、`PARSE_ERROR`。
3. 抖音和小红书只做样式和状态占位：
   - “公开可见，但需浏览器会话”。
   - 不生成采集任务。
4. 如后续确实要做浏览器辅助采集：
   - 独立 Worker。
   - 单独数据目录。
   - 不进入主后端进程。
   - 需要人工登录、可撤销、可清理。

## 7. 风险边界

当前项目不建议实现：

- 平台验证码绕过。
- 平台动态签名逆向的内置实现。
- 批量高频扫描用户主页。
- 默认保存用户账号 Cookie。
- 粉丝列表、关注列表等关系链采集。
- 大规模评论采集。

允许的保守能力：

- 低频读取公开用户基础资料。
- 保存字段快照。
- 保存失败状态和退避时间。
- 标注来源不稳定。
- 为后续人工授权或浏览器会话留扩展点。

## 8. 最终判断

如果目标是“指定任意用户，监控名称、粉丝数等公开数据”：

- **微博可以尝试，但必须低频、退避、标注不稳定。**
- **抖音不建议直接做后端定时采集，除非接受长期维护动态参数和浏览器环境。**
- **小红书暂缓，除非产品明确接受浏览器会话和登录态成本。**

对当前 `BiliMonitor` 最务实的路线：

```text
B站继续作为稳定主线
  -> 微博低频公开资料 PoC
  -> 抖音/小红书只做待接入状态
  -> 后续如需要，再独立做 Browser Assisted Worker
```
